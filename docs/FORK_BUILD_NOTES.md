# MacroQuest Private Fork — Build Runbook (x64 EQ Live)

Reproduction notes for getting this fork building cleanly from a fresh clone,
and the rationale behind the fork-specific build fixes. Preset/CMake is the
single supported build path; the legacy MSBuild solution path is intentionally
disabled (see Phase 1.6).

## Goal & environment

- **Target:** x64, EQ Live. `eqlib` is on the `live` branch (`EXPANSION_LEVEL_SOR`) — do not change its branch.
- **Build path:** `cmake --preset live` then `cmake --build build --config Release`.
- **vcpkg:** vendored in-tree at `contrib/vcpkg` (`VCPKG_ROOT=${sourceDir}/contrib/vcpkg`),
  triplet `x64-windows-static`, manifest mode → installs to `build/vcpkg_installed`.
  Default registry = `macroquest/vcpkg`.
- **Toolchain:** Visual Studio 2022 (v143), Windows SDK 10.x.

## Reproduce from a clean clone

```powershell
git clone https://github.com/WenzelJD/MacroQuest.git
cd MacroQuest
git submodule update --init --recursive       # eqlib (live), vcpkg, contrib
cmake --preset live                            # installs deps incl. cpr/pe-parse; builds loader (default ON)
cmake --build build --config Release
```

**Expected artifacts in `build/bin/Release/`:** `MacroQuest.exe`, `MQ2Main.dll`,
`eqlib.dll`, `imgui-64.dll`, `crashpad_handler.exe`, `luarocks.exe`, and 14 plugin
DLLs under `plugins/`.

> If you ever see `C1083: 'google/protobuf/runtime_version.h'` again, it is stale
> `.pb.*` in `src/` (the legacy protobuf generator sneaking back). Delete those
> files — do **not** touch include paths. See Phase 1 and the protobuf desync note.

## Phase 1 — Get MQ2Main + full Release green (commit `046cfb09`)

**Headline error (`C1083: runtime_version.h`) root cause:** stale, git-ignored
`.pb.cc/.pb.h` in the *source tree* (`src/routing`, `src/login`, `src/plugins/lua`),
generated by a protobuf **6.33.4** protoc. Consumers do `#include "Routing.pb.h"`,
so the quoted include resolved to the sibling stale 33.4 header (which references
`runtime_version.h`) instead of the correct **3.21.12** header CMake generates into
`build/`. It was never an include-path problem.

Fixes:

1. **Delete stale source-tree `.pb.*`** — git-ignored (`.gitignore` `*.pb.*`),
   regenerated into `build/` by `protobuf_generate`.
   ```
   find src -name "*.pb.cc" -o -name "*.pb.h"   # then remove them
   ```
2. **`src/Common.cmake`**: `/std:c++17` → `/std:c++20`. The per-target flag overrode
   the global C++20 standard (LanguageStandard wins) and broke
   `std::unordered_set::contains` in `Network.cpp`.
3. **`include/mq/contrib/protobuf/ProtobufLibs.h`**: removed the abseil + utf8_range
   `#pragma comment(lib, ...)` auto-links; kept only `libprotobuf`/`libprotobufd`.
   protobuf 3.21.12 is self-contained; abseil/utf8 are not installed, so those
   pragmas caused `LNK1104: absl_base.lib` in routing/login/main/lua.
4. **`src/imgui/CMakeLists.txt`**: added the missing `imanim/im_anim.cpp`,
   `im_anim_demo.cpp`, `im_anim_doc.cpp`, `im_anim_usecase.cpp` (+ headers). The
   vcxproj→CMake conversion dropped them → unresolved `iam_*`/`ImAnim*` in MQ2Main.
5. **`src/main/CMakeLists.txt`**: config-aware crashpad sublib override.
   `crashpadConfig.cmake` resolves `common/client/util/base` with a non config-aware
   `find_library()` that baked the *debug* libs into Release → `LNK2038` (debug CRT
   `_ITERATOR_DEBUG_LEVEL`/`MTd`). Re-point per config:
   ```cmake
   if(TARGET crashpad)
       set(_crashpad_lib_rel "${CMAKE_BINARY_DIR}/vcpkg_installed/${VCPKG_TARGET_TRIPLET}/lib")
       set(_crashpad_lib_dbg "${CMAKE_BINARY_DIR}/vcpkg_installed/${VCPKG_TARGET_TRIPLET}/debug/lib")
       set(_crashpad_sublibs "")
       foreach(_crashpad_l common client util base)
           list(APPEND _crashpad_sublibs
               "$<IF:$<CONFIG:Debug>,${_crashpad_lib_dbg}/${_crashpad_l}.lib,${_crashpad_lib_rel}/${_crashpad_l}.lib>")
       endforeach()
       set_target_properties(crashpad PROPERTIES
           INTERFACE_LINK_LIBRARIES "${_crashpad_sublibs};ZLIB::ZLIB")
   endif()
   ```
   (Done at the project level, not in the port, to avoid invalidating crashpad's
   cached ABI and forcing a full gn rebuild.)
6. **`tools/build_scripts/protoc/protobuf.targets`**: disabled the legacy MSBuild
   `_ProtocolBuffer` target (`Condition="false"`). That target was the only place
   the legacy path ran protoc (6.33.4) with `--cpp_out` into the source tree — the
   stale-file generator. The preset build uses `protobuf_generate` → `build/` and
   does not import this file, so it is unaffected.

## Phase 2 — Protobuf version desync (understanding; no code change)

Two vcpkg installs coexist by design of two build paths, and they disagree:

- **Manifest** (`build/vcpkg_installed`, used by the CMake/preset build):
  protobuf **3.21.12**, resolved from the `macroquest/vcpkg` registry baseline.
  Its protoc (`build/vcpkg_installed/x64-windows/tools/protobuf/protoc.exe`) emits
  into `build/`. `contrib/vcpkg/ports` is NOT on this build's `overlay-ports` list.
- **Classic** (`contrib/vcpkg/installed`, used by the old MSBuild path): protobuf
  **6.33.4**, from the overlay port `contrib/vcpkg/ports/protobuf` (pinned 6.33.4).
  Its protoc emits `.pb.*` into the source tree.

The CMake build is internally consistent on 3.21.12. The 33.4 files only ever came
from the legacy MSBuild path, which Phase 1.6 disables. If protobuf is ever bumped
to 22+/33.4 (e.g. adding `contrib/vcpkg/ports` to the manifest overlay-ports),
restore the abseil/utf8 pragmas in `ProtobufLibs.h` and regenerate headers.

## Phase 3 — Enable the loader / `MacroQuest.exe` (commit `c8d4bb13`)

`MQ_BUILD_LAUNCHER` flipped ON in `CMakeLists.txt`; the `# TODO: Fix the loader
cmake` marker removed. `src/loader` fixes:

1. **`find_package(crashpad)`** guarded with `if(NOT TARGET crashpad)`.
   `crashpadConfig.cmake` does an unguarded `add_library(crashpad INTERFACE)`; with
   `src/main` already finding it, a second call errored ("target already exists").
   Reusing the existing target also inherits the Phase 1.5 CRT fix.
2. Dropped the per-target **`/std:c++17`** override (let Common's `/std:c++20` apply).
3. Removed the stale **`${VCPKG_IncludeStatic}/crashpad`** include (undefined var →
   bogus `/crashpad`); crashpad headers come from the `crashpad::crashpad` target.
4. Replaced the empty **`${LoaderDependencies*}`** (these were MSBuild properties,
   not CMake variables) with the Win32 system libs they carried:
   `d3d11 Setupapi imm32 Secur32 Ws2_32 Wldap32 version`. The vcpkg/project libs
   from the original list are already provided by the `find_package` targets.
5. **`curl::curl` → `CURL::libcurl`** (correct vcpkg target name).
6. **`src/loader/portfile.cmake`**: removed stale `vcpkg_install_empty_package()`
   (unknown command) that broke the loader overlay meta-port install;
   `set(VCPKG_POLICY_EMPTY_PACKAGE enabled)` alone is correct (matches the other
   overlay ports). Note: the overlay generator only writes `portfile.cmake` if it
   does not already exist, so this stale committed copy was never regenerated.
7. **`src/loader/PostOffice.h`**: added `#include <windows.h>` (after
   `ServerPostOffice.h`, so asio/winsock is included first) — the header declares
   functions taking `HWND` but consumers included it before any Win32 header,
   leaving `HWND` undefined (`C2065` + cascade).

> Enabling the loader on an existing build dir required `-DMQ_BUILD_LAUNCHER=ON`
> once, because CMake `option()` does not override a value already cached as OFF.
> The cache now holds ON and the source default is ON, so future
> `cmake --preset live` runs build the loader.

  real injector test still requires the game.
- `contrib/crashpad/lib_release_mt/util.lib` (~55 MB) is over GitHub's 50 MB soft
  limit; candidate for Git LFS or exclusion.
- `contrib/vcpkg/ports/protobuf` is still pinned to 6.33.4 and the classic
  `contrib/vcpkg/installed` tree still exists. Both are inert for the preset build
  but are cleanup candidates if you want to fully collapse the dual-vcpkg topology.
